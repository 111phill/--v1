# 大学生成绩管理系统 — 设计报告

---

# 第一章 项目概述

## 1.1 项目介绍

### 一、建设背景

当前高校教务管理正经历从"半手工 + Excel 碎片化"向"全流程数字化"的转型期，但成绩管理环节仍普遍存在以下痛点：

1. **数据孤岛严重**：教务处使用教务系统，任课教师使用 Excel 录分，辅导员通过微信/邮件收集成绩预警信息，三者数据互不相通。一次期末成绩的汇总往往需要跨 3~4 个系统/工具的反复导入导出。
2. **权限粒度粗糙**：传统教务系统的权限模型仅区分"管理员"与"普通用户"，无法满足"同一名教师对本人授课班级有写权限、对其他班级仅有读权限"、"辅导员仅查看本年级不及格名单"等精细化需求。
3. **并发瓶颈突出**：每学期末，全校数万名学生同时查分、教师集中录分，传统单体架构在高峰期响应时间飙升至 10 秒以上，甚至发生锁表宕机。
4. **审计追溯缺失**：成绩修改缺乏完整的操作日志链（谁、何时、从何值改为何值、原因），一旦出现成绩争议，难以快速定位责任。
5. **移动端体验差**：绝大多数教务系统未做响应式适配，学生无法在手机上便捷查分、教师无法在平板上完成录分。

### 二、建设意义

本系统以"一套数据、多角色协同、全流程可追溯"为核心理念，将成绩录入、审核、发布、查询、统计、预警等环节整合为统一平台。通过容器化部署降低运维门槛，通过响应式前端覆盖 PC/平板/手机全终端，通过细粒度 RBAC 实现数据安全与操作便捷的平衡。

### 三、核心目标

| 目标 | 具体描述 |
|------|----------|
| **全角色覆盖** | 支持管理员、教师、学生三类角色，每类角色拥有独立的功能集与数据可见范围 |
| **全业务闭环** | 覆盖课程管理 → 成绩录入 → 审核发布 → 查询统计 → 学业预警的完整链路 |
| **高并发承载** | 期末查分高峰期，系统应支撑 ≥ 500 QPS 的并发查询，P99 延迟 ≤ 500ms |
| **数据安全** | 基于 JWT + RBAC + 操作日志三重机制，确保成绩数据的机密性、完整性与可追溯性 |
| **一键部署** | 提供 Docker Compose 编排文件，实现前后端 + 数据库的一键启动 |

## 1.3 主要技术与工具简介

### 1.3.1 Spring Boot 4

Spring Boot 4（基于 Spring Framework 7）是 2025 年底发布的重大版本，核心优势包括：

- **虚拟线程（Virtual Threads）原生支持**：基于 JDK 21 Project Loom，Tomcat 请求处理线程从平台线程切换为虚拟线程。在成绩查询这类 I/O 密集型场景中，虚拟线程可大幅降低线程上下文切换开销，同等硬件下并发吞吐量提升 2~3 倍。
- **AOT 编译与 GraalVM 原生镜像**：支持将 Spring 应用编译为原生可执行文件，启动时间从秒级降至毫秒级，内存占用降低 60% 以上，特别适合容器化部署场景。
- **增强的 Actuator 可观测性**：内置 Micrometer Tracing，集成 OpenTelemetry，可导出到 Prometheus + Grafana 实现全链路监控。
- **声明式 HTTP 接口（`@HttpExchange`）**：替代 RestTemplate，以接口 + 注解方式定义 HTTP 调用，代码更简洁、类型更安全。

### 1.3.2 Vue 3（Composition API）

Vue 3 的 Composition API 是本项目前端架构的核心选择：

- **逻辑复用与代码组织**：通过 `composable`（组合式函数）将成绩查询、Token 管理、权限校验等可复用逻辑抽离为独立模块，避免 Options API 中 `mixins` 的命名冲突与来源不清问题。
- **`<script setup>` 语法糖**：编译时语法糖，减少样板代码，组件定义更简洁。
- **响应式系统升级（Proxy 代理）**：Vue 3 使用 ES6 Proxy 替代 Vue 2 的 `Object.defineProperty`，支持对数组索引、Map/Set、动态属性的响应式追踪，在处理成绩列表的复杂 CRUD 操作时 API 更自然。
- **Tree-shaking 友好**：按需引入，打包体积较 Vue 2 减小约 40%。

### 1.3.3 MyBatis + Mapper 接口

本系统选择 MyBatis 而非 JPA 的原因：

- **SQL 完全可控**：成绩统计涉及多表联查（学生-课程-成绩-班级-专业）和聚合函数（GPA 计算、排名），复杂的 SQL 需要手写优化。MyBatis 的 XML Mapper 允许开发者精确控制每一条 SQL，包括索引提示、子查询优化等。
- **Mapper 接口模式**：`@Mapper` 接口 + XML 映射文件的组合，接口方法名即语义，XML 中编写具体 SQL，兼顾 Java 类型安全与 SQL 灵活性。
- **动态 SQL**：`<if>`/`<foreach>`/`<choose>` 标签天然支持"按课程名/学号/班级/学期"等多条件组合筛选，避免拼接字符串的繁琐与注入风险。

### 1.3.4 Docker & Docker Compose

- **环境一致性**：消除"开发环境正常、部署环境报错"的经典问题。前端（Nginx + 静态资源）、后端（JAR）、数据库（MySQL）三服务通过一个 `docker-compose.yml` 编排，一键启动。
- **资源隔离与弹性伸缩**：每个服务独立容器，CPU/内存限制可配置。演示时可使用 `docker compose up -d` 完成全部部署。
- **数据持久化**：MySQL 数据目录通过 Volume 映射到宿主机，容器重启不丢失数据。

### 1.3.5 GitHub

- **版本控制**：基于 Git Flow 分支模型，`main` 为发布分支，`develop` 为开发分支，功能开发在 `feature/*` 分支进行。
- **CI/CD**：通过 GitHub Actions 在 Push / PR 时自动执行 Maven 构建 + 单元测试 + Docker 镜像构建，保证主分支始终可部署。

# 第二章 需求分析

## 2.1 功能需求

本节从角色视角出发，采用"角色 → 用例 → 详细描述"的三层结构，完整覆盖高校成绩管理业务。

### 2.1.1 学生端功能

学生是系统的主要查询者，权限限定为"仅查看本人数据"。

| 编号 | 用例名称 | 详细描述 |
|------|----------|----------|
| S-01 | **个人成绩查询** | 按学期筛选，展示课程名称、学分、平时成绩、期末成绩、总评成绩、绩点、班级排名，支持表格导出为 Excel |
| S-02 | **GPA 总览** | 展示累计 GPA（加权平均）、已修学分、不及格学分，以学期为横轴的 GPA 趋势折线图（ECharts） |
| S-03 | **成绩单下载** | 生成带防伪水印的 PDF 成绩单（含学校 Logo、二维码校验入口） |
| S-04 | **学业预警查看** | 查看系统自动生成的个人预警信息（如累计不及格学分 ≥ 20 触发退学预警），标记已读/未读 |
| S-05 | **个人信息维护** | 修改手机号、邮箱等联系方式，不可修改学号、姓名等身份关键字段 |

### 2.1.2 教师端功能

教师的数据权限遵循"仅操作本人授课班级"原则，由管理员在排课时绑定教师-课程-班级关系。

| 编号 | 用例名称 | 详细描述 |
|------|----------|----------|
| T-01 | **授课班级查看** | 展示当前学期本人授课的课程列表，每门课程显示选课人数、成绩录入进度（已录/总人数） |
| T-02 | **成绩录入与提交** | 以 Excel 风格的表格界面批量录入成绩。支持：(a) 从 Excel 文件导入（自动匹配学号，Apache POI 解析）；(b) 手动逐条录入；(c) 暂存草稿与正式提交两步操作，提交后锁定不可修改 |
| T-03 | **成绩统计分析** | 对本人授课班级的成绩进行统计：最高/最低/平均分、各分数段人数分布（柱状图）、及格率、标准差 |
| T-04 | **成绩修改申请** | 已提交的成绩不可直接修改，需填写修改原因并提交申请，由管理员审核后执行修改（全流程留痕） |
| T-05 | **教学班名单导出** | 导出含学号、姓名、平时/期末/总评成绩的完整成绩登记表（Excel） |

### 2.1.3 管理员端功能

管理员拥有全局管理权限，负责系统的基础数据维护与业务审核。

| 编号 | 用例名称 | 详细描述 |
|------|----------|----------|
| A-01 | **用户管理** | 批量导入（Excel）或单个创建学生/教师账号，支持启用/禁用账号、重置密码、分配角色 |
| A-02 | **专业与班级管理** | 维护"学院 → 专业 → 班级"三级树形结构，支持增删改查 |
| A-03 | **课程管理** | 创建课程（课程编号、名称、学分、学时、考核方式），绑定授课教师与开课班级 |
| A-04 | **学期管理** | 设置当前学期（如 2025-2026-2），控制成绩录入的起止时间窗口 |
| A-05 | **成绩审核与发布** | 审核教师提交的成绩修改申请，通过后执行修改并记录操作日志。控制成绩发布状态（草稿/已发布），发布后学生端可见 |
| A-06 | **学业预警规则配置** | 设置预警规则（如：单学期不及格学分 ≥ 12 → 黄色预警，累计不及格学分 ≥ 20 → 橙色预警，累计 ≥ 30 → 红色退学预警），支持按专业差异化配置 |
| A-07 | **操作日志审计** | 查看全系统操作日志（时间、操作人、IP、操作类型、操作前值、操作后值、备注），支持按时间范围与操作类型筛选 |
| A-08 | **数据统计看板** | 仪表盘展示：各学院平均绩点排名、不及格率 Top 10 课程、本学期成绩录入完成率、近 4 学期 GPA 走势等（ECharts 可视化） |

## 2.2 非功能需求

### 2.2.1 性能需求

| 指标 | 要求 | 验证方式 |
|------|------|----------|
| **并发查询** | 成绩查询接口在 500 并发用户下，平均响应时间 ≤ 200ms，P99 ≤ 500ms | JMeter 压力测试，逐步加压至 500 线程 |
| **批量导入** | 教师导入 200 条成绩的 Excel 文件，处理时间 ≤ 3 秒 | 上传标准测试文件计时 |
| **数据库连接池** | HikariCP 连接池，`maximumPoolSize = 20`，连接超时 30 秒 | Spring Boot 配置 + Actuator `/metrics` 监控 |
| **前端首屏加载** | 首页（仪表盘）首次加载 ≤ 2 秒（4G 网络），二次加载（缓存）≤ 500ms | Vite 打包后 Lighthouse 审计 |
| **虚拟线程** | Tomcat 请求处理线程池切换为虚拟线程，支撑 1000+ 并发连接 | `spring.threads.virtual.enabled = true` |

### 2.2.2 安全性需求

| 需求项 | 具体方案 |
|--------|----------|
| **身份认证** | 用户登录成功后，后端签发 JWT（RS256 非对称签名），含 `userId`、`role`、`exp`（过期时间 2 小时）。前端将 Token 存储于 `httpOnly` Cookie（防 XSS 窃取），每次请求通过 Axios 拦截器自动携带 |
| **Token 刷新** | 在 JWT 过期前 30 分钟，前端主动调用 `/api/auth/refresh` 接口续期，避免用户在使用中因 Token 过期需重新登录 |
| **RBAC 权限控制** | 后端基于 Spring Security 的 `@PreAuthorize` 注解实现方法级权限控制。例如：`@PreAuthorize("hasRole('TEACHER') and @courseService.isInstructor(#courseId)")` 确保教师仅能操作本人课程 |
| **SQL 注入防护** | MyBatis 全部使用 `#{}` 参数占位符（预编译），禁止 `${}` 拼接用户输入。成绩查询的多条件动态 SQL 通过 `<if>` 标签安全构建 |
| **密码存储** | 使用 BCrypt（`spring-security-crypto`）对密码进行加盐哈希，不可逆存储 |
| **HTTPS 传输** | 生产环境强制 HTTPS，Nginx 反向代理配置 SSL 证书（演示环境可使用自签名证书） |
| **操作日志** | 所有成绩的增/改/删操作均通过 AOP 切面自动记录：操作人、IP、时间、操作类型、实体类型、操作前 JSON 快照、操作后 JSON 快照。日志表按月分区 |

### 2.2.3 可用性需求

| 需求项 | 具体方案 |
|--------|----------|
| **响应式布局** | 基于 Element Plus 的 24 栅格系统，设定三个断点：`xs (<768px)` 手机竖屏—单列布局、折叠菜单；`sm (768px~992px)` 平板—双列布局；`md (≥992px)` 桌面—标准后台管理三栏布局（侧栏 + 顶栏 + 内容区） |
| **前端异常处理** | Axios 响应拦截器统一处理 HTTP 错误码：401 → 跳转登录页并清除本地 Token；403 → 提示"无操作权限"；500 → 提示"服务器内部错误，请联系管理员"并展示错误追踪 ID；网络超时 → 提示"网络连接超时，请检查网络" |
| **后端全局异常处理** | `@RestControllerAdvice` + `@ExceptionHandler` 统一捕获异常：(a) `MethodArgumentNotValidException` → 返回字段级校验错误详情（HTTP 400）；(b) `AccessDeniedException` → 返回统一禁止访问响应（HTTP 403）；(c) 未捕获异常 → 生成 TraceId 并记录 ERROR 日志，返回通用错误信息（HTTP 500） |
| **数据校验** | 前后端双重校验。前端：Element Plus Form 组件的 `rules` 校验（学号格式、成绩范围 0~100、必填项等）；后端：`@Valid` + Jakarta Bean Validation（`@NotNull`、`@Min`、`@Max`）在 Controller 层拦截非法输入 |
| **数据库备份** | MySQL 每日凌晨 3:00 通过 `mysqldump` 自动备份，保留最近 7 天的备份文件 |
| **优雅关闭** | Spring Boot 配置 `server.shutdown=graceful`，收到 SIGTERM 后等待现有请求处理完毕（最多 30 秒）再关闭，避免成绩提交中途被中断导致数据不一致 |

### 2.2.4 可维护性需求

| 需求项 | 具体方案 |
|--------|----------|
| **API 文档** | 集成 SpringDoc OpenAPI（Swagger 3），自动生成 REST API 文档，访问 `/swagger-ui.html` 可在线调试 |
| **日志规范** | 使用 SLF4J + Logback，按包名分级：`com.grade.controller` → DEBUG（记录请求参数），`com.grade.service` → INFO（记录业务操作），`com.grade.security` → WARN（记录鉴权失败）。日志输出到文件并按天滚动，保留 30 天 |
| **配置外部化** | 所有环境相关配置（数据库连接、JWT 密钥、文件上传路径）通过 `application-{profile}.yml` 或环境变量注入，Docker 部署时由 `docker-compose.yml` 的 `environment` 段覆盖 |

# 第三章 系统总体设计

## 3.1 系统总体结构

本系统采用**前后端分离**的四层架构，前端与后端通过 HTTP/HTTPS 协议以 JSON 格式交换数据，二者可独立开发、独立构建、独立部署。

### 3.1.1 架构分层与交互流程

```
┌─────────────────────────────────────────────────────────────┐
│                     客户端 (Client)                          │
│  浏览器: Vue 3 + Element Plus + ECharts                     │
│  Nginx 托管静态资源 + 反向代理 API 请求到后端网关               │
└──────────────────────┬──────────────────────────────────────┘
                       │  HTTP/HTTPS (JSON)
                       ▼
┌─────────────────────────────────────────────────────────────┐
│              表现层 (Presentation Layer)                     │
│  - Vue Router 路由管理（含路由守卫 authGuard）                │
│  - Axios 请求/响应拦截器（Token 注入、401 跳转、错误提示）     │
│  - Pinia 状态管理（userStore、gradeStore）                   │
│  - Element Plus 响应式 UI 组件                               │
└──────────────────────┬──────────────────────────────────────┘
                       │  RESTful API (JSON)
                       ▼
┌─────────────────────────────────────────────────────────────┐
│              业务层 (Business Layer)                         │
│  Spring Boot 4 后端服务（内嵌 Tomcat + 虚拟线程）             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────────┐ │
│  │ 安全过滤层│→│Controller│→│ Service  │→│   Mapper     │ │
│  │ JWT Filter│  │ @Valid   │  │ @Trans-  │  │  接口调用    │ │
│  │ + RBAC   │  │ 参数校验  │  │ actional │  │              │ │
│  └──────────┘  └──────────┘  └──────────┘  └─────────────┘ │
```

### 3.1.3 一次请求的完整交互流程

以"学生查询本学期成绩"为例，描述各层间的协作过程：

1. **表现层**：学生在 Vue 页面选择"2025-2026-2"学期，点击查询。Pinia `gradeStore` 调用 Axios 发送 `GET /api/grades?semesterId=5`。Axios 请求拦截器自动从 Cookie 中提取 JWT Token 并注入 `Authorization: Bearer <token>` 头。
2. **业务层-安全过滤**：`JwtAuthenticationFilter` 拦截请求，验证 Token 签名（RS256 公钥验签），解析出 `userId=2021001`、`role=STUDENT`，构建 `UsernamePasswordAuthenticationToken` 存入 `SecurityContextHolder`。
3. **业务层-Controller**：`GradeController.getMyGrades()` 接收请求，`@PreAuthorize("hasRole('STUDENT')")` 校验角色通过。
4. **业务层-Service**：`GradeService.getByStudentAndSemester(userId, semesterId)` 执行业务逻辑 — 从 `SecurityContextHolder` 获取当前用户 ID（而非信任前端传参，防止越权），调用 Mapper 查询。
5. **持久层**：`GradeMapper.selectByStudentAndSemester(userId, semesterId)` 执行 XML 中预定义的 SQL（三表 JOIN：`t_grade` + `t_course` + `t_semester`），返回 `List<GradeVO>`。
6. **表现层**：Axios 接收到 JSON 响应，`gradeStore` 更新响应式数据，Vue 组件自动重新渲染成绩表格。

### 3.2.2 业务层设计

#### 一、权限校验业务流程

系统采用"JWT 无状态认证 + Spring Security 过滤器链 + 方法级注解"的三层权限模型：

```
请求到达 → JwtAuthenticationFilter
                │
                ├── 无 Authorization 头 → 继续传递（由后续 Filter 拦截 → 401）
                │
                ├── Token 签名无效/过期 → 返回 401
                │
                └── Token 合法 → 解析 claims 得到 userId + role
                                  │
                                  └── 构建 Authentication 对象
                                      存入 SecurityContextHolder
                                                │
                                                ▼
                                    方法执行前触发 @PreAuthorize
                                                │
                              ┌─────────────────┼─────────────────┐
                              │                  │                  │
                        hasRole('xxx')   hasAuthority('xxx')   自定义 SpEL
                              │                  │                  │
                              ▼                  ▼                  ▼
                         角色匹配通过      权限字符串匹配      调用 Bean 方法
                                                                │
                                                    @courseService
                                                    .isInstructor(courseId)
                                                        │
                                                  查询 t_course 表
                                                  确认教师是否绑定此课程
```

**关键代码逻辑（`JwtAuthenticationFilter.java`）：**

1. 从请求头提取 `Authorization: Bearer <token>`
2. 使用 `JwtParser` + RS256 公钥验证签名与过期时间
3. 从 Payload 提取 `userId`、`username`、`role`
4. 构建 `JwtAuthenticationToken(userId, username, role, authorities)` 存入 `SecurityContextHolder`
5. 所有后续业务代码可通过 `SecurityUtils.getCurrentUserId()` 获取当前用户 ID，避免信任前端传入的 `userId` 参数

#### 二、成绩计算业务流程

**（1）总评成绩计算**

成绩计算在教师提交成绩时由后端执行（不信任前端计算值）：

```
总评成绩 = 平时成绩 × 平时占比 + 期末成绩 × 期末占比
```

平时占比与期末占比由课程表（`t_course`）的 `usual_ratio` 和 `exam_ratio` 字段定义（如 30% : 70%）。计算逻辑位于 `GradeServiceImpl.calculateTotalScore()` 方法中：

```java
public BigDecimal calculateTotalScore(BigDecimal usualScore, 
        BigDecimal examScore, Course course) {
    return usualScore.multiply(course.getUsualRatio())
            .add(examScore.multiply(course.getExamRatio()))
            .setScale(1, RoundingMode.HALF_UP);
}
```

**（2）绩点（GPA）换算**

绩点换算采用国内高校通用的 4.0 分制映射规则：

| 总评成绩区间 | 绩点 |
|-------------|------|
| [90, 100] | 4.0 |
| [85, 90) | 3.7 |
| [82, 85) | 3.3 |
| [78, 82) | 3.0 |
| [75, 78) | 2.7 |
| [72, 75) | 2.3 |
| [68, 72) | 2.0 |
| [64, 68) | 1.5 |
| [60, 64) | 1.0 |
| [0, 60) | 0.0 |

累计 GPA 使用加权平均公式：

```
累计GPA = Σ(课程绩点 × 课程学分) / Σ(已修课程学分)
```

GPA 计算在 `GradeServiceImpl.calcGpa()` 方法中通过 MyBatis 的聚合查询一次性完成，避免逐条 Java 循环计算。

**（3）班级排名计算**

班级排名在成绩发布后异步计算（`@Async` + 虚拟线程），SQL 使用 MySQL 窗口函数：

```sql
SELECT student_id, course_id, total_score,
       RANK() OVER (PARTITION BY course_id, class_id 
                    ORDER BY total_score DESC) AS class_rank
FROM t_grade
WHERE semester_id = #{semesterId} AND status = 'PUBLISHED'
```

#### 三、成绩录入与提交流程

```
教师选择课程 + 班级 → 系统加载该班学生名单
            │
            ├── 方式一：页面表格逐条录入
            │          │

            │              └── 逐行调用 saveDraft()，汇总成功/失败数
            │
            └── 点击"正式提交"
                        │
                        ▼
                GradeService.submitGrades(courseId, classId, semesterId)
                        │

                        │        （SELECT teacher_id FROM t_course_class
```

### 3.2.4 Mapper XML 示例（核心查询）

为确保第三章内容自包含，以下给出 `GradeMapper.xml` 中两个代表性 SQL 的完整实现：

**（1）学生成绩查询（三表 JOIN）**

```xml
<select id="selectByStudentAndSemester" resultMap="GradeVOMap">
    SELECT g.id, g.total_score, g.gpa, g.status,
           c.id AS course_id, c.name AS course_name, 
           c.credit, c.usual_ratio, c.exam_ratio,
           g.usual_score, g.exam_score
    FROM t_grade g
    INNER JOIN t_course c ON g.course_id = c.id
    WHERE g.student_id = #{studentId}
      AND g.semester_id = #{semesterId}
    ORDER BY c.id
</select>
```

**（2）成绩统计（聚合查询）**

```xml
<select id="calcStatistics" resultType="GradeStatistics">
    SELECT
        COUNT(*) AS total_count,
        MAX(total_score) AS max_score,
        MIN(total_score) AS min_score,
        ROUND(AVG(total_score), 1) AS avg_score,
        ROUND(STDDEV(total_score), 2) AS std_deviation,
        COUNT(CASE WHEN total_score &gt;= 60 THEN 1 END) AS pass_count,
        ROUND(COUNT(CASE WHEN total_score &gt;= 60 THEN 1 END) 
              * 100.0 / COUNT(*), 1) AS pass_rate
    FROM t_grade
    WHERE course_id = #{courseId}
      AND semester_id = #{semesterId}
      AND class_id = #{classId}
      AND status = 'SUBMITTED'
</select>
```

```
┌─────────────────────────────────────────────────┐
│  POST /api/v1/auth/login                        │
│  Body: { username, password, role }             │
└──────────────────┬──────────────────────────────┘
                   │

    AuthController.login(LoginRequest req)
                   │

                   │

                   │        ├── 用户不存在 → 抛出 BadCredentialsException("用户名或密码错误")

                   │        │
                   │        ├── 不匹配 → 抛出 BadCredentialsException("用户名或密码错误")
                   │        │    （统一错误信息，防止用户名枚举攻击）
                   │        │
                   │        └── 匹配 → 继续
                   │
                   ├── 3. 校验 user.status == "ACTIVE"
                   │        │
                   │        └── 已禁用 → 抛出 DisabledException("账号已被禁用")
                   │
                   ├── 4. 校验 user.role == req.role
                   │        │

                   │

                            │
                            ├── 加载 RS256 私钥（classpath:jwt/private.pem）
                            │
                            ├── 构建 Payload:
                            │   { "sub": "1",             // userId

                            │   SignedJWT = new SignedJWT(header, payload)
```

**关键后端代码（`JwtTokenProvider.java` 核心方法）：**

```java
@Component
public class JwtTokenProvider {

    private RSAPrivateKey privateKey;
    private RSAPublicKey publicKey;
    @Value("${jwt.expiration}") private long expiration;

    @PostConstruct
    public void init() {
        // 从 PEM 文件加载 RSA 密钥对
        this.privateKey = RsaKeyLoader.loadPrivateKey("classpath:jwt/private.pem");
        this.publicKey = RsaKeyLoader.loadPublicKey("classpath:jwt/public.pem");
    }

    public String generateToken(Long userId, String username, String role) {
        Instant now = Instant.now();
        return Jwts.builder()
            .subject(userId.toString())
            .claim("username", username)
            .claim("role", role)
            .issuedAt(Date.from(now))
            .expiration(Date.from(now.plusSeconds(expiration)))
            .signWith(privateKey, Jwts.SIG.RS256)
            .compact();
    }

    public Claims validateAndParse(String token) {
        return Jwts.parser()
            .verifyWith(publicKey)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }
}
```

**路由守卫（`router/index.ts` 伪代码）：**

```
router.beforeEach((to, from, next) => {
    const userStore = useUserStore()

    // 白名单路由直接放行
    if (to.meta.requiresAuth === false) {
        // /login 页面
        next()
        return
    }

    // 检查 Token 是否存在
    if (!userStore.token) {
        // 未登录 → 跳转登录页，携带 redirect 参数
        next({ path: '/login', query: { redirect: to.fullPath } })
        return

    // 检查 Token 是否即将过期（距过期 < 30 分钟）
    if (isTokenExpiringSoon(userStore.token, 30 * 60)) {

        userStore.refreshToken()

    // 校验角色是否有权访问目标路由
    if (to.meta.roles && !to.meta.roles.includes(userStore.role)) {
        // 无权限 → 跳转到对应角色的首页
        next({ path: getDefaultRoute(userStore.role) })
```

**（2）`GradeServiceImpl.java` — 核心业务逻辑**

```java
@Service
@Transactional
public class GradeServiceImpl implements GradeService {

    private final GradeMapper gradeMapper;
    private final CourseMapper courseMapper;
    private final SemesterMapper semesterMapper;
    private final AuditLogMapper auditLogMapper;

    public BatchResult batchSave(Long teacherId, Long courseId,
            Long classId, Long semesterId, String action,
            List<GradeItem> grades) {

        // === 前置校验 ===

        // 1. 教师授课权校验：确认此教师被分配到此课程-班级
        int count = courseMapper.countTeacherBinding(
            teacherId, courseId, classId, semesterId);
        if (count == 0) {
            throw new AccessDeniedException("您不是此课程的授课教师");
        }

        // 2. 时间窗口校验：当前是否在成绩录入时间段内
        Semester semester = semesterMapper.selectById(semesterId);
        LocalDateTime now = LocalDateTime.now();
        if (now.isBefore(semester.getGradeEntryStart()) ||
            now.isAfter(semester.getGradeEntryEnd())) {
            throw new BusinessException("当前不在成绩录入时间窗口内");
        }

        // 3. 防止重复提交：检查是否已正式提交（SUBMITTED 状态不可改）
        int submittedCount = gradeMapper.countByStatus(
            courseId, classId, semesterId, "SUBMITTED");
        if (submittedCount > 0) {
            throw new BusinessException("成绩已提交，不可修改。" +
                "如需修改请发起修改申请");
        }

        // === 核心处理 ===

        Course course = courseMapper.selectById(courseId);
        int savedCount = 0;

        for (GradeItem item : grades) {
            // 计算总评成绩（后端计算，不信任前端值）
            BigDecimal totalScore = calculateTotalScore(
                item.getUsualScore(),
                item.getExamScore(), course);

            Grade grade = Grade.builder()
                .studentId(item.getStudentId())
                .courseId(courseId)
                .semesterId(semesterId)
                .classId(classId)
                .usualScore(item.getUsualScore())
                .examScore(item.getExamScore())
                .totalScore(totalScore)
                .status(action)       // DRAFT 或 SUBMITTED
                .entryTeacherId(teacherId)
                .build();

            // INSERT ... ON DUPLICATE KEY UPDATE
            // 唯一键: (student_id, course_id, semester_id)
            gradeMapper.insertOrUpdate(grade);
            savedCount++;
        }

        // 操作日志（AOP 自动记录，此处仅示意）
        // auditLogMapper.insert(new AuditLog(
        //     teacherId, "GRADE_BATCH_SAVE",
        //     "courseId=" + courseId + ", action=" + action,
        //     savedCount + " 条"
        // ));

        return new BatchResult(savedCount,
            action.equals("SUBMITTED") ? "成绩提交成功，已锁定" : "成绩暂存成功");
    }

    // 成绩计算（private 方法，仅本类调用）
    private BigDecimal calculateTotalScore(BigDecimal usual,
            BigDecimal exam, Course course) {
        return usual.multiply(course.getUsualRatio())
            .add(exam.multiply(course.getExamRatio()))
            .setScale(1, RoundingMode.HALF_UP);
    }
}
```

3. 后端在单次 SQL 中完成三表 JOIN（`t_grade` + `t_course` + `t_semester`），返回含课程名、学分、绩点的完整 VO 列表
4. 前端 `ECharts` 折线图渲染 GPA 走势，表格展示明细

**后端核心 SQL（`GradeMapper.xml` — GPA 趋势查询）：**

```xml
<select id="selectGpaTrend" resultType="GpaTrend">
    SELECT
        s.id AS semester_id,
        s.name AS semester_name,
        ROUND(SUM(g.gpa * c.credit) / SUM(c.credit), 2) AS semester_gpa,
        ROUND(
            SUM(SUM(g.gpa * c.credit)) OVER (
                ORDER BY s.start_date
                ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
            ) / SUM(SUM(c.credit)) OVER (
                ORDER BY s.start_date
                ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
            ), 2) AS cumulative_gpa
    FROM t_grade g
    INNER JOIN t_course c ON g.course_id = c.id
    INNER JOIN t_semester s ON g.semester_id = s.id
    WHERE g.student_id = #{studentId}
      AND g.status = 'PUBLISHED'
    GROUP BY s.id, s.name, s.start_date
    ORDER BY s.start_date
</select>
```

### 6.1.2 后端 Dockerfile 设计

**设计思路：**

1. **多阶段构建（Multi-stage Build）**：第一阶段使用 `maven:3.9-eclipse-temurin-21` 镜像完成源码编译打包，第二阶段仅保留 JRE 运行环境。两个阶段分离使得最终镜像体积从 600+ MB 压缩至约 200 MB。
2. **非 root 用户运行**：创建 `gradeapp` 用户执行 JAR，遵循最小权限原则，防止容器逃逸后的权限提升。
3. **优雅关闭支持**：`ENTRYPOINT` 使用 `exec java` 而非脚本包装，确保 JVM 能正确接收 `docker stop` 发出的 SIGTERM 信号，配合 `server.shutdown=graceful` 实现优雅关闭。
4. **分层缓存优化**：先 COPY 依赖较少的 `app.jar`，利用 Docker 构建缓存加速重复构建。

```dockerfile
# ============ 第一阶段：构建阶段 ============
FROM maven:3.9-eclipse-temurin-21 AS builder

WORKDIR /build
# 先复制 pom.xml，利用 Docker 缓存层加速依赖下载
COPY pom.xml .
RUN mvn dependency:go-offline -B

# 复制源码并打包（跳过测试，测试在 CI 阶段已执行）
COPY src ./src
RUN mvn package -DskipTests -B

# ============ 第二阶段：运行阶段 ============
FROM eclipse-temurin:21-jre

# 创建非 root 用户
RUN groupadd -r gradeapp && useradd -r -g gradeapp gradeapp

WORKDIR /app

# 从构建阶段复制 JAR
COPY --from=builder /build/target/grade-system-*.jar app.jar

# 创建数据目录并授权
RUN mkdir -p /app/logs /app/uploads && chown -R gradeapp:gradeapp /app

USER gradeapp

# 优雅关闭：exec 形式确保 JVM 收到 SIGTERM
ENTRYPOINT ["java", "-jar", "app.jar"]

# 健康检查
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
    CMD curl -f http://localhost:8080/actuator/health || exit 1

EXPOSE 8080
```

### 6.1.3 前端 Dockerfile 设计

**设计思路：**

1. **Node 构建 + Nginx 托管**：第一阶段使用 `node:20-alpine` 运行 `vite build` 生成静态资源，第二阶段使用 `nginx:1.26-alpine` 托管。
2. **自定义 Nginx 配置**：SPA 模式需将所有非 `/api/` 请求回退到 `index.html`，防止刷新页面出现 404；后端 API 通过 `proxy_pass` 反向代理。
3. **Gzip 压缩**：开启 `gzip_static`，Vite 构建时预先生成 `.gz` 文件，Nginx 直接返回压缩版本，减少传输体积约 70%。

```dockerfile
# ============ 第一阶段：构建阶段 ============
FROM node:20-alpine AS builder

WORKDIR /build
COPY package.json package-lock.json ./
RUN npm ci

COPY . .
RUN npm run build     # 执行 vite build，输出到 dist/

# ============ 第二阶段：运行阶段 ============
FROM nginx:1.26-alpine

# 删除默认 Nginx 配置
RUN rm /etc/nginx/conf.d/default.conf

# 复制自定义 Nginx 配置
COPY nginx.conf /etc/nginx/conf.d/

# 复制构建产物到 Nginx 静态目录
COPY --from=builder /build/dist /usr/share/nginx/html

EXPOSE 80

HEALTHCHECK --interval=30s --timeout=3s \
    CMD wget -qO- http://localhost/ || exit 1
```

**对应 `nginx.conf`：**

```nginx
server {
    listen 80;
    server_name _;

    root /usr/share/nginx/html;
    index index.html;

    # Gzip 压缩
    gzip on;
    gzip_static on;
    gzip_types text/plain text/css application/json application/javascript text/xml;

    # API 反向代理到后端容器
    location /api/ {
        proxy_pass http://grade-backend:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 30s;
    }

    # SPA 前端路由回退
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

### 6.1.4 Docker Compose 编排文件设计

**设计思路：**

1. **服务依赖与启动顺序**：MySQL 容器需先于后端容器启动，使用 `depends_on` + `healthcheck` 条件（`condition: service_healthy`）确保 MySQL 完全就绪后才启动后端。
2. **网络隔离**：创建自定义 `bridge` 网络 `grade-net`，三容器通过服务名互访（如后端使用 `grade-mysql:3306`），不暴露不必要的端口到宿主机。
3. **环境变量注入**：敏感配置（数据库密码、JWT 密钥路径）通过 `environment` 和 `secrets` 注入，不硬编码在镜像中。
4. **数据持久化**：MySQL 数据目录和上传文件目录通过命名卷（named volume）映射，`docker compose down` 不会丢失数据。

```yaml
version: '3.9'

services:
  # ==================== MySQL 数据库 ====================
  mysql:
    image: mysql:8.0
    container_name: grade-mysql
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASS:-Grade@2026Sec}
      MYSQL_DATABASE: grade_system
      MYSQL_CHARSET: utf8mb4
      MYSQL_COLLATION: utf8mb4_unicode_ci
    volumes:
      - mysql-data:/var/lib/mysql
      - ./sql/init.sql:/docker-entrypoint-initdb.d/01-init.sql:ro
    ports:
      - "3306:3306"
    networks:
      - grade-net
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  # ==================== Spring Boot 后端 ====================
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    image: grade-backend:latest
    container_name: grade-backend
    restart: unless-stopped
    depends_on:
      mysql:
        condition: service_healthy              # 等待 MySQL 健康后才启动
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://grade-mysql:3306/grade_system?useUnicode=true&characterEncoding=utf8mb4
      SPRING_DATASOURCE_USERNAME: root
      SPRING_DATASOURCE_PASSWORD: ${MYSQL_ROOT_PASS:-Grade@2026Sec}
      JWT_PRIVATE_KEY_PATH: /app/config/private.pem
      JWT_PUBLIC_KEY_PATH: /app/config/public.pem
      SERVER_PORT: 8080
    volumes:
      - upload-data:/app/uploads
      - ./config/jwt:/app/config:ro              # JWT 密钥只读挂载
      - backend-logs:/app/logs
    ports:
      - "8080:8080"
    networks:
      - grade-net
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  # ==================== Nginx 前端 ====================
  nginx:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    image: grade-nginx:latest
    container_name: grade-nginx
    restart: unless-stopped
    depends_on:
      - backend
    ports:
      - "80:80"
    networks:
      - grade-net
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost/"]
      interval: 30s
      timeout: 3s
      retries: 3

# ==================== 卷与网络 ====================
volumes:
  mysql-data:                                    # MySQL 数据持久化
  upload-data:                                   # 上传文件（Excel 模板、成绩单 PDF）
  backend-logs:                                  # 后端日志持久化

networks:
  grade-net:
    driver: bridge
```

## 7.2 遇到的问题及解决方法

### 问题一：Spring Boot 4 虚拟线程下 `ThreadLocal` 导致的内存泄漏

**问题描述**：在成绩查询接口中，使用 `ThreadLocal` 存储当前请求的 `TraceId` 用于日志链路追踪。切换到虚拟线程后，JMeter 长时间压测出现内存持续增长，约 30 分钟后 OOM（OutOfMemoryError）。

**原因分析**：虚拟线程由 JVM 内部的 `ForkJoinPool` 管理，线程对象会被频繁创建和回收，但不会调用传统线程池的 `ThreadLocal.remove()` 清理逻辑。`ThreadLocal` 中存储的对象无法被 GC 回收，虚拟线程的高频创建/销毁导致泄漏速度远超平台线程。

**解决方法**：

```java
// ❌ 错误写法：ThreadLocal 在虚拟线程下泄漏
public class TraceContext {
    private static final ThreadLocal<String> TRACE_ID = new ThreadLocal<>();
    public static void setTraceId(String id) { TRACE_ID.set(id); }
    public static String getTraceId() { return TRACE_ID.get(); }
}

// ✅ 正确写法：使用 ScopedValue（JDK 21+ 替代 ThreadLocal）
public class TraceContext {
    public static final ScopedValue<String> TRACE_ID = ScopedValue.newInstance();
}

// 在 Filter 中使用 ScopedValue.where() 绑定
ScopedValue.where(TraceContext.TRACE_ID, traceId)
    .run(() -> chain.doFilter(request, response));
```

`ScopedValue` 是 JDK 21 为虚拟线程设计的不可变作用域变量，生命周期严格绑定于 `run()` 方法的执行范围，虚拟线程被回收时自动清理，消除了 `ThreadLocal.remove()` 的手动管理需求。

### 问题二：Vue 3 `watchEffect` 在成绩表格编辑中的死循环

**问题描述**：教师成绩录入页面使用 `watchEffect` 监听 `gradeTable` 的变化以自动计算总评成绩预览。当成绩表格有 45 条数据时，修改某条记录的平时成绩，`watchEffect` 回调中重新计算总评成绩并写回 `gradeTable`，触发新一轮 `watchEffect`，导致无限循环，页面卡死。

**原因分析**：`watchEffect` 在其回调内部触发了被监听对象的变更，形成"修改 → 触发 → 修改 → 触发"的死循环。且 `watchEffect` 会追踪回调内访问的所有响应式属性，粒度过粗。

**解决方法**：改用 `computed` 在渲染时按需计算，而非通过 `watchEffect` 主动修改源数据。

```typescript
// ❌ 错误写法：watchEffect 中修改源数据导致死循环
watchEffect(() => {
    gradeTable.value.forEach(g => {
        g.previewScore = g.usualScore * 0.3 + g.examScore * 0.7
        // 修改 gradeTable 触发 watchEffect 再次执行 → 死循环
    })
})

// ✅ 正确写法：computed 纯计算，不修改源数据
const displayTable = computed(() =>
    gradeTable.value.map(g => ({
        ...g,
        previewScore: g.usualScore != null && g.examScore != null
            ? (g.usualScore * 0.3 + g.examScore * 0.7).toFixed(1)
            : null
    }))
)
```

模板中绑定 `displayTable` 而非 `gradeTable`，`computed` 仅在 `gradeTable` 实际变化时重新计算，无副作用，从根源上消除了死循环。

### 问题三：Docker Compose 中 MySQL 启动未就绪导致后端连接失败

**问题描述**：`docker compose up -d` 执行后，`grade-backend` 容器报错 `CommunicationsException: Communications link failure`，随后退出。虽然 `docker-compose.yml` 中已配置 `depends_on: mysql`，但问题仍然出现。

**原因分析**：`depends_on` 默认仅等待容器"启动"（即进程存在），不等待容器内服务"就绪"。MySQL 容器启动后，`mysqld` 进程需执行初始化（创建数据库、执行 `init.sql`、刷新权限表），此过程约需 10~30 秒。在此期间后端尝试连接数据库必然失败。Spring Boot 默认的连接重试机制不足以覆盖此时间窗口。

**解决方法**：在 v3.9+ Compose 文件中使用 `condition: service_healthy` 结合 MySQL 的 `healthcheck`：

```yaml
# docker-compose.yml 关键配置
services:
  mysql:
    image: mysql:8.0
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s          # ← 关键：给予 30s 初始化宽限期

  backend:
    depends_on:
      mysql:
        condition: service_healthy  # ← 等 MySQL healthcheck 通过后才启动
```

同时在后端 `application.yml` 中增加连接重试配置：

```yaml
spring:
  datasource:
    hikari:
      initialization-fail-timeout: 60000   # 等待连接池初始化最多 60s
```

二者配合，后端容器最多等待 60 秒（30s 初始化 + 3 次 healthcheck × 10s），MySQL 就绪后再建立连接池，彻底解决了启动时序问题。
